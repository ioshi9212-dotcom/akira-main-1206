# Dynamic Context Loading Policy

## Назначение

Этот файл фиксирует правила точечной загрузки контекста для Akira Main 1206.

Он заменяет опасную схему полного контекста и деления на огромные блоки `core / characters / memory`.

Контекст должен быть точным, а не большим.

## Главный принцип

Перед сценой нужно определить:

1. где мы сейчас;
2. когда мы сейчас;
3. кто физически рядом;
4. кто говорит / действует / наблюдает;
5. что сделал игрок;
6. какие темы реально затронуты;
7. какие отношения / знания / state могут измениться.

Только после этого грузить файлы.

## Базовый минимальный контекст

Всегда нужен короткий обязательный минимум:

- session context / `current_state`;
- `turn-contract`, если доступен;
- `gpt/engine_prompt.md`;
- `gpt/scene_format.md`;
- `gpt/context_loading_policy.md`;
- текущий scene-файл или `current_scene_anchor`;
- `calendar/story_calendar.md` для текущей даты / текущего окна;
- последние 1–3 релевантные записи scene history или summary, если доступны.

Этот минимум не должен превращаться в полный дамп всего репозитория.

## Turn-contract важнее общего списка

Если API отдаёт `turn-contract`, он главный источник списка загрузки.

`turn-contract.required_files` отвечает на вопрос: какие файлы нужны именно этому ходу.

Если общий prompt говорит «учесть тему», а turn-contract не включает конкретный файл, файл можно не тянуть без причины.

Исключение:

- `gpt/engine_prompt.md`;
- `gpt/scene_format.md`;
- `gpt/context_loading_policy.md`;
- session context / `current_state`.

## Активные персонажи

Карточки персонажей грузятся только для:

- Акиры;
- active characters;
- nearby characters;
- speaking characters;
- observing characters, если их наблюдение влияет на сцену;
- mentioned characters, если упоминание требует точности.

Не грузить всех главных персонажей заранее.

Если Райден не рядом и не участвует в сцене, его карточка не нужна только потому, что он важен для будущего.

Если Юна должна появиться до 5 сентября, но сейчас не медблок и не её окно, её карточку не нужно тянуть.

## Условные персонажи

Условный персонаж становится активным только после выполнения условия.

Пример: Райден в стартовой сцене.

До условия:

- не тянуть его карточку как active;
- не писать звук, силуэт, голос или реакцию;
- не давать ему знание сцены;
- не делать его причиной действий других персонажей.

После условия:

- добавить в active/nearby;
- загрузить карточку;
- загрузить knowledge, если сцена касается знаний / прошлого / скрытых реакций;
- загрузить relationship pair, если есть социальное взаимодействие.

## Knowledge

Не грузить весь knowledge-state каждый ход.

Нужно вытаскивать:

- знания активных персонажей;
- знания персонажа, который произносит важную реплику;
- знания персонажа, чьё прошлое / имя / слух / тайна обсуждается;
- сведения о том, знает ли Акира имя / лицо / статус персонажа;
- сведения о том, кто был свидетелем события.

Hidden lore не является knowledge персонажа.

## Relationships

Relationships грузить точечно:

- пары активных персонажей;
- пары, где один персонаж прямо обсуждается;
- пары, где есть текущий конфликт, ревность, флирт, обещание, долг или продолжение линии;
- пары с open_threads по текущей сцене.

Не грузить весь relationships без причины.

Для важной пары нужны не только цифры, но и:

- `status`;
- `notes`;
- `memory`;
- `open_threads`;
- `triggers`;
- `last_interaction`;
- `behavior_next`;
- preferences.

## Preferences

Preferences грузить только если сцена социальная и поведение персонажа зависит от реакции на действия Акиры / окружающих.

Примеры:

- Акира спорит с Райденом → preference Райдена нужна.
- Акира просто идёт по коридору, Райден не рядом → preference Райдена не нужна.
- Лиана ревнует / конкурирует → preference Лианы нужна.
- Хару реагирует на риск / шутку / защиту → preference Хару может быть нужна.

## Topic canon

Canon-файлы грузятся только по теме.

### База / Восточный сектор

Если сцена касается Восточной базы, общежития, столовой, медблока, двора, фонтана, трензала, бассейна, рейдов или быта базы:

- `canon/east_sector_base.md`

### Энергия / бой / тренировка

Если сцена касается энергии, боя, тренировки, перегруза, контроля, допусков:

- `canon/energy_system.md`

### Пространство Акиры / Эхо / карман / переход

Если сцена касается пространства Акиры, водопада как ощущения, кармана, поглощения, схлопывания, Эхо рядом с Акирой:

- `canon/akira_space_energy_mechanics.md`

### Браслеты / блокаторы / маскировка сигнатуры

Если сцена касается браслета, скрытия энергии, того, почему Акиру не чувствуют, защитного поля или ограничителей:

- `canon/energy_limiters_and_signature_masking.md`

### Кайросы / гибриды / материк / договор

Если сцена касается кайросов, гибридов, материка, беловолосых, договора, запрета отношений, Ирэя, Эммы, Самуэля как линии контроля:

- `canon/kairos_public_and_hidden.md`
- `canon/world_public_lore.md`, если нужен публичный слой мира.

### Hidden world lore

Если нужна скрытая причинность Эхо, баланса или Акиры как источника:

- `canon/world_hidden_lore.md`

Но hidden lore не должен попадать в прямую речь NPC, если персонаж этого не знает.

## Calendar

`calendar/story_calendar.md` грузится по текущей дате / ближайшему окну.

Не нужно грузить будущие месяцы, если сцена не делает таймскип.

При таймскипе проверять:

- есть ли обязательные события между текущим и целевым временем;
- есть ли условные события;
- выполнены ли условия;
- кого можно вводить;
- что запрещено до даты.

Календарь не даёт NPC знания будущего.

## Context budget

Если контекст слишком большой, не делить его механически на три части.

Вместо этого:

1. убрать неактивных персонажей;
2. убрать будущие события, не относящиеся к текущей дате;
3. заменить длинную scene_history на summary + последние 1–3 события;
4. взять только нужные relationship pairs;
5. взять только нужные knowledge slices;
6. оставить hidden lore вне prompt, если он не нужен для причинности текущего хода;
7. убрать canon-файлы, тема которых не затронута.

## Если файл не найден

Если не найден обязательный файл:

- `current_state`;
- `gpt/scene_format.md`;
- текущий scene-файл;
- карточка active/speaking персонажа;
- нужный knowledge/relationship для сцены раскрытия;

нельзя писать сцену уверенно.

Если не найден опциональный файл, можно продолжить только если сцена не зависит от него.

## Запрещено

- Не грузить полный контекст каждый ход.
- Не делить контекст на огромные `part/core`, `part/characters`, `part/memory` как обязательную схему.
- Не грузить всех персонажей ради безопасности.
- Не грузить весь relationships целиком без причины.
- Не использовать hidden lore как знания NPC.
- Не считать историю чата источником истины вместо state.
- Не продолжать сцену из головы, если current_state недоступен.
- Не вводить future character только потому, что он есть в календаре.

## Короткая формула

Сначала текущая сцена.

Потом активные персонажи.

Потом нужные отношения.

Потом topic canon.

Не весь мир. Только нужный слой.
